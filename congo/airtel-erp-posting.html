How the pieces connect

S&D sends Invoice and Receipt data into staging interface tables (mkt_erp_*_intf). A concurrent program picks up records with PROCESS_FLAG = 'N', posts them across the CG_ASIPROD database link into ERP‑side staging tables (asi_snd_*_stg), then flips POSTED_TO_ERP to 'Y'.     Section 05

    Procedures & Calling Methods

    Primary Sale and Primary Sale Return are the only flows that need a manual trigger — everything else calls the ERP posting itself.

    How it works

    • Once the Primary Sale is done and the invoice is generated, the Order is completed.
    • The procedure for a completed Primary Sale and Primary Sale Return must be called manually.
    • For POS Sale, POS Sale Return, Payment Receipt, Payment Receipt Reversal, Credit Memo and Debit Memo, the application itself calls the ERP posting — there is no procedure to call.
    Shared tax formula — applied to every row above
    DIVISOR = 1 + (VAT% / 100) + (VAT% / 100 × CA% / 100)
    BASE = (QTY × UNIT_PRICE_TAX_INCLUDED) / DIVISOR
    VAT_AMOUNT = BASE × VAT% / 100
    CA_AMOUNT = VAT_AMOUNT × CA% / 100

    Calling methods

    Both procedures accept the same two call shapes — one specific order, or a date range sweep.

    PRC_INSERT_ERP_SELLIN_STAGING — Sales

    By order ID
    BEGIN
    PRC_INSERT_ERP_SELLIN_STAGING(
        p_order_id => '2349004133'
    );
END;
    By date range
    BEGIN
    PRC_INSERT_ERP_SELLIN_STAGING(
        p_from_date => TO_DATE('2026-05-01', 'YYYY-MM-DD'),
        p_to_date   => TO_DATE('2026-05-30', 'YYYY-MM-DD')
    );
END;

    PRC_INSERT_ERP_SELLIN_RETURN_STAGING — Sales Return

    By order ID
    BEGIN
    PRC_INSERT_ERP_SELLIN_RETURN_STAGING(
        p_order_id => '2349004133'
    );
END;
    By date range

    Operational SQL — backfill & reprocessing

    Ad‑hoc queries for finding orders that never reached the ERP interface, and for re‑arming already‑staged rows so the concurrent posting program picks them up again with a corrected GL date. Run the find query first, then paste the returned keys into the IN (…) lists before running the updates.

    1 · Find COMPLETED orders not posted to ERP

    POS_ORDERS with no matching row in MKT_ERP_INV_INTF
    SELECT * FROM POSUSER.POS_ORDERS po
WHERE NOT EXISTS (
    SELECT 1
    FROM esbuser.MKT_ERP_INV_INTF meii
    WHERE meii.reference_no = po.ORDER_ID
)
AND trunc(po.ORDER_DATE) >= trunc(sysdate - 20)   --Change the date accordingly
AND po.STATUS = 'COMPLETED';

    2 · Re‑arm staged rows with the latest GL date

    AR Invoice staging — reset TRANSACTION_DATE
    UPDATE apps.asi_snd_ar_invoice_stg@CG_ASIPROD
SET PROCESS_FLAG     = 'N',
    TRANSACTION_DATE = TO_DATE('2026-06-01 00:00:00', 'YYYY-MM-DD HH24:MI:SS')
WHERE TRANSACTION_NO IN ( …transaction_no list… );
    AR Receipt staging — reset GL_DATE
    UPDATE apps.asi_snd_ar_receipt_stg@CG_ASIPROD
SET PROCESS_FLAG = 'N',
    GL_DATE      = TO_DATE('2026-07-01 00:00:00', 'YYYY-MM-DD HH24:MI:SS')
WHERE RECEIPT_NUMBER IN ( …receipt_number list… );

    Setting PROCESS_FLAG = 'N' puts the rows back in the queue; the corrected date drives the period they post into. Populate each IN (…) list and adjust the dates before executing.
